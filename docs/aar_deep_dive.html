Architecture · 01

Design Principles

Thin Core Loop

loop.py is a single coroutine doing exactly three things: call the provider, execute tool calls, append events to the session. Everything else is delegated to sibling modules.

Typed Event Model

Every interaction is a Pydantic model. Events are serializable, carry timing data, and are the single source of truth. No raw dicts in the event stream.

Provider-Agnostic

The loop works with any Provider ABC implementation. Swapping from Anthropic to Ollama requires changing one config field. Never import a provider outside agent/providers/.

Safe by Default

Path restrictions, command deny-lists, and approval gates are built in and always active. Interactive modes enable a workspace sandbox automatically. Defaults can only be hardened, not relaxed, via CLI.

Modular Transports

The same Agent class runs from CLI, TUI, web API, ACP, or embedded in code. Transports only handle I/O — never business logic. They wire events to UI, nothing more.

Fully Observable

Every event has a timestamp and is persisted to JSONL. session_metrics() reconstructs full telemetry from history — no live agent needed.

Dependency rule

Core never imports providers. Providers never import tools. Tools never bypass safety. Transports never contain business logic. One direction only.

ACP is now a package

transports/acp/ is split into three focused modules: stdio.py for Zed (SDK), http.py for REST clients, and common.py for shared types. Old single-file acp.py is gone.

WSL management

safety/wsl_manager.py handles WSL2 distro download, setup, and lifecycle — keeping sandbox.py focused purely on command execution.

Core Loop · 02

The Loop in Plain English

LOOP — repeat until done:

  ① Guard checks
       Cancelled?     → stop immediately
       Timed out?     → stop
       Too many steps → stop

  ② Trim context  (if sliding_window is on)
       Messages exceed token budget → drop oldest

  ③ Call the LLM
       Send: messages + available tools + system prompt
       Retry automatically on network errors
       Receive: text,  tool_calls,  stop_reason,  token usage

  ④ Check budget
       Tokens or cost over the limit? → stop

  ⑤ Handle the response

       if tool_calls present:
           Same calls as last step × 3? → stuck guard fires → stop
           Otherwise: run each tool safely → loop back to ①

       else (no tool calls):
           stop_reason = max_tokens?
               Still have retries? → inject "keep going…" → loop back to ①
               No retries left?    → treat as finished

           stop_reason = end_turn?
               → DONE ✓

Three rules of the loop

The loop does exactly three things and nothing else:

  1. Call the provider
  2. Execute tool calls
  3. Append events to the session

Everything else (retries, token counting, event formatting) lives in sibling modules.

Event order matters

ToolCall is emitted before AssistantMessage in the same step. This lets session.to_messages() bundle tool-use blocks correctly for the next API call — matching Anthropic's expected message format.

Streaming teardown

The stream loop runs inside try/finally — always emitting StreamChunk(finished=True) even if the provider throws. TUI and SSE consumers never hang.

Persistence

State is set on the Session object before returning, and persisted to JSONL. Sessions can be resumed — they restart at RUNNING from the last event.

Guardrails · 04

Mechanical Safety Nets

Catch runaway loops and truncated responses without relying on the LLM to self-regulate. Purely mechanical — no prompt heuristics, no dynamic system-prompt mutation.

① Max-tokens Recovery

When the provider truncates output (stop_reason = "max_tokens"), the loop injects a continuation prompt and loops again — up to a configurable limit.

stop_reason = "max_tokens"
└─ recovery_count < max_tokens_recoveries?
     YES → inject "Continue from where you left off…"
           (tagged: internal, not shown in UI)
           loop again
     NO  → treat as END_TURN → exit loop

② Repetition Circuit-Breaker

Detects identical tool-call patterns repeating and aborts before burning budget in an infinite spin.

each step: observe_tool_calls(session, tool_calls)
  → build signature: sorted names + args
    (values truncated at 200 chars)
  → same as last step?
      repeated_tool_steps += 1
  → different?
      repeated_tool_steps = 0

is_stuck()?
  repeated_tool_steps ≥ max_repeated_tool_steps
    YES → emit ErrorEvent, state = ERROR

③ Budget Proximity

near_budget() fires before hard limits. Transports use it for visual warnings. Hard enforcement is in the loop itself.

tokensremaining ≤ reserve_tokens (512)
costremaining ≤ cost_limit × 0.1

Config

"guardrails": {
  "max_tokens_recoveries":   2,
  "max_repeated_tool_steps": 3,
  "reserve_tokens":          512,
  "reserve_cost_fraction":   0.1
}

Counters live in session.metadata["guardrails"] — persisted and restored on session reload.

Guardrails · 04b

Tuning by Use Case

Use caseSuggested adjustment
Cloud provider, tight budgetRaise reserve_cost_fraction to 0.15–0.20 to warn earlier
Long-reasoning models (DeepSeek, o3)Raise max_tokens_recoveries to 3–4 — truncation is more likely
Autonomous / high-step runsKeep max_repeated_tool_steps at 3; lower only if you trust the model
Headless / unattendedLower max_repeated_tool_steps to 2 for faster abort
Local models, no cost limitreserve_cost_fraction has no effect when cost_limit = 0
Interactive TUIDefaults are good — user can observe and intervene

Continuation prompt (internal)

The injected prompt on max-tokens recovery is tagged data["reason"] = "max_tokens_recovery" and is not shown in the UI. The counter is bumped; if it reaches the recovery limit, the loop exits normally as if it were end_turn.

Why the signature includes values

The repetition signature hashes tool names and argument key=value pairs (truncated at 200 chars). Calling read_file on different paths is not repetition — only truly identical call sets trigger the guard.

Why mechanical, not prompt-based?

Prompt instructions like "don't loop forever" fail when the model is confused or hallucinating. Mechanical guardrails are deterministic — they fire regardless of model behavior, every single time, with no exceptions.

Tool System · 05b

Tool Error Result Format

Every error path returns a ToolResult with a stable, machine-readable category tag in the output.

Error [<category>]: human-readable message
CategoryWhen it fires
unknown_toolTool name not found in the registry
no_handlerTool registered without a callable handler
invalid_argumentsArguments fail JSON-schema validation
blockedSafety policy returned DENY
deniedHuman declined an approval-gated call
timeoutHandler exceeded ToolConfig.command_timeout
exceptionHandler raised an unhandled exception

Match on category, not message

# ✅ stable across versions
if result.output.startswith("Error [blocked]"):
    ...

# ❌ fragile — message text changes
if "policy denied" in result.output:
    ...

Executor wraps everything

ToolExecutor adds duration_ms and enforces max_output_chars on every result — success or error. The loop never receives raw exceptions from tool execution.

Safety · 06

Safety: Three Layers

1
SafetyPolicy.check_tool()
Evaluates in order: read-only mode → approval requirements → path rules → denied_paths → allowed_paths → command rules → denied_commands → default ALLOW
2
PermissionManager
Handles ASK decisions via the ApprovalCallback. Caches APPROVED_ALWAYS so the user isn't prompted twice for the same tool.
3
Sandbox
OS-level isolation for shell commands. Local / Linux Landlock / Windows Job Object / WSL2 distro.

Always-denied paths

/etc/shadow**/.env **/credentials**/*.pem **/.ssh/****/.aws/** **/.npmrc**/.pypirc

Always-denied commands

rm -rf /mkfs shutdown:(){:\|:&};: chmod 777curl|sh nc -ehistory -c

Can be extended via config — never reduced via CLI flags alone.

Approval results

APPROVEDThis specific call only
DENIEDThis call is blocked
APPROVED_ALWAYSCached — all future calls to this tool
Safety · 06b

Per-Transport Safety Defaults

aar chat / aar tui — Interactive

Workspace sandbox ON by default:

SettingDefaultEffect
--require-approvalonPrompts before every write or shell cmd
--restrict-to-cwdonFile tools can only access cwd/**

Two-layer defense: file tools restricted by allowed_paths; bash bypasses that restriction so it requires human approval instead.

aar run — Automation

Permissive by default — designed for trusted, scripted workflows:

SettingDefault
--require-approvaloff
--restrict-to-cwdoff

aar serve — Web API

No approval callback by default — any ASK decision is auto-denied (no human present). Override per-request via the body:

Sandboxes · 07

Sandbox Modes

Selected by SafetyConfig.sandbox.mode. All sandboxes return stdout+stderr capped at max_output_chars.

local LocalSandbox

Direct asyncio.create_subprocess_exec. No isolation at all — for trusted environments only. Maximum compatibility.

linux LinuxSandbox

Landlock LSM restricts writes to the workspace directory. ulimit -v memory cap. Restricted environment variables. Kernel-enforced, no root required.

windows WindowsSubprocessSandbox

Windows Job Object enforces memory and process count caps. Process runs at Low Integrity Level — blocked from writing to most system locations by the OS.

wsl WslDistroSandbox

Runs commands inside a dedicated WSL2 distro: wsl -d <distro> -- sh -c <cmd>. Strongest isolation on Windows. Distro lifecycle managed by wsl_manager.py.

auto Auto Mode

Detects the platform at runtime:

Linux→ linux (Landlock)
Windows + WSL config→ wsl
Windows→ windows (Job Object)
Fallback→ local

Recommended for cross-platform configs.

Streaming fallback

If a stream closes without a terminal done=True delta, _consume_stream() logs at WARNING and falls back to END_TURN or TOOL_USE. SSE transports never hang.

End to End

Complete Data Flow

User / Transport │ (string or multimodal content) ▼ Agent.run() ├─ Session.append(UserMessage) ▼ ───── LOOP ITERATION ───────────────────────────────────────────────────────────────── │ ├─ ① Guard checks cancel / timeout / max_steps → exit early if triggered │ ├─ ② Context trim sliding_window: trim_to_token_budget(messages) │ ├─ ③ Provider (provider_runner.py) │ provider.complete() or provider.stream() │ retry with exponential back-off on recoverable errors │ emit ProviderMeta (timing + token usage) │ ├─ ④ Budget check token_budget / cost_limit → BUDGET_EXCEEDED if exceeded │ ├─ ⑤ tool_calls? │ YES ─┬─ observe_tool_calls() → is_stuck()? → ERROR if repeated ≥ threshold │ ├─ for each tool_call: │ │ emit ToolCall │ │ SafetyPolicy.check_tool() → ALLOW / DENY / ASK │ │ if ASK → ApprovalCallback (terminal prompt or auto-deny) │ │ sandbox.run() → emit ToolResult (duration_ms + truncated output) │ └─ emit AssistantMessage (bundles tool_use blocks) → loop back to ① │ │ NO ─┬─ max_tokens + recovery budget left? │ │ → inject internal continuation prompt → loop back to ① │ └─ end_turn (or max_tokens, no budget) │ emit AssistantMessage → done = True ───── END ITERATION ────────────────────────────────────────────────────────────────── │ ▼ Session.state = COMPLETED (or CANCELLED / ERROR / MAX_STEPS / TIMED_OUT / BUDGET_EXCEEDED) │ ├─ JSONL persisted: one line per event │ └─ Transport event callbacks received every event in real time
